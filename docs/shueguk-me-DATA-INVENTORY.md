# 슈국 데이터 지형도 — 기존 도구 점검 결과

> `shueguk-me` 2·3단계(활동 모으기 + 포인트/계급)를 위한 사전 조사.
> 기존 도구들이 **학생을 어떻게 식별하고, 데이터를 어디에 기록하는지** 점검한 결과.
> _(조사 범위: 이 세션에서 열람 가능한 `shueguk-hub`, `shueguk-clinic`만. 나머지는 별도 저장소라 미확인.)_

---

## 1. 시스템은 "여러 저장소 + 각자의 백엔드"로 흩어져 있음

`shueguk-hub`의 메뉴 카드들은 **런처일 뿐**, 실제 도구는 대부분 별도 저장소에 있고 **각자 다른 Google Apps Script(=각자 다른 스프레드시트)** 를 백엔드로 쓴다.

| 도구 | 학생용 진입점 | 백엔드 | 이 세션에서 확인? |
|---|---|---|---|
| 클리닉 | `shueguk-clinic` (index.html) | Apps Script (JSONP) `AKfycbw8e-054…` | ✅ 확인됨 |
| 주말 실전모의고사(OMR) | `shueguk-hub/omr.html` → iframe | Apps Script 웹앱 `AKfycbyUHMdC…` (UI를 GAS가 렌더) | ⚠️ 부분 |
| 어휘 테스트 | `shueguk-voca` (별도 repo) | 자체 (links.html, teacher-dashboard) | ❌ 범위 밖 |
| 문법 테스트 | `shueguk-gramma` (별도 repo) | 자체 | ❌ 범위 밖 |
| H WORK | `shueguk-h-work` (별도 repo) | 자체 | ❌ 범위 밖 |
| 지필 리포트 | `shueguk-report` (별도 repo) | 미사용 | ❌ 범위 밖 |

---

## 2. 학생 식별 키가 도구마다 다름 (⚠️ 핵심 난제)

| 도구 | 학생 식별 방식 |
|---|---|
| **클리닉** | 이름 + 학교 + **전화 뒤 4자리** + 클리닉 시간 |
| **로그인(설계안)** | 이름 + **학부모 전화 가운데 4자리** |
| OMR / 어휘 / 문법 / H WORK | 미확인 (GAS 코드 또는 별도 저장소 안) |

→ **클리닉은 "전화 뒤 4자리", 로그인 설계는 "학부모 전화 가운데 4자리"** 로 이미 다름.
→ 각 도구가 제각각이면 *"이 활동들이 모두 같은 학생"* 임을 시스템이 알 수 없어 **포인트 합산 불가**.

---

## 3. 함의 — 포인트/계급(3단계)의 진짜 작업

데이터가 **여러 스프레드시트에 흩어져** 있고 **식별 키가 제각각**이므로, 포인트를 모으려면 다음 중 하나가 필요하다:

- **(A) 공통 학생 ID로 통일** — 명단(로스터)의 `학생코드`를 기준 삼아, 모든 도구가 결과에 같은 키를 기록하도록 점진적으로 맞춤. *(가장 깔끔, 장기적으로 권장)*
- **(B) 중앙 집계 시트** — 각 도구의 Apps Script에 "결과를 공통 ID로 중앙 시트에 한 줄 추가" 코드를 더해, 흩어진 데이터를 한곳에 모음.
- **(C) 브릿지에서 사후 매칭** — 이름+전화 등으로 여러 시트를 대조해 합침. *(동명이인·키 불일치에 취약, 임시방편)*

권장: **로스터의 `학생코드`를 단일 기준점으로 삼고 (A)+(B) 병행.**

---

## 4. 단계별 영향 요약

- **1단계(로그인 + 성적표 + 공지):** ✅ 영향 없음. 로스터 + Notion만으로 독립적으로 가능.
- **2단계(활동 모아 보기):** 각 도구의 결과 시트를 읽어야 함 → 식별 키 통일이 전제.
- **3단계(포인트/계급):** 위 통일 위에서만 정확히 합산 가능.
